# Phase4_2 总结

## 本阶段目标

本阶段的目标不是直接做知乎正文抓取，而是先验证第四阶段“企业增强”最小闭环是否可用：

1. 能否稳定请求知乎搜索接口
2. 能否把搜索结果写入 `company_enriched`
3. 能否基于搜索结果做第一轮企业风险证据提取
4. 能否确认“阅读全文”这条路是否值得继续投入

## 本阶段完成内容

### 1. 新增独立入口

已新增：

- `company_enrich.py`

当前职责：

- 从 `jobs_cleaned` 中筛出：
  - `need_company_check = 1`
  - `company_check_status = 'pending'`
- 调用知乎搜索接口
- 将结果写入 `company_enriched`
- 同步更新 `jobs_cleaned.company_check_status`

### 2. 新增知乎搜索客户端

已新增：

- `crawler/zhihu_client.py`

当前实现方式：

- 不走 Playwright
- 不读搜索页 DOM
- 不点击页面元素
- 直接请求知乎 `search_v3` 接口
- 使用完整 Cookie + 接近真实浏览器的请求头

当前确认：

- 只保留少量 cookie 键不够
- 需要尽量接近浏览器真实请求
- `x-zse-96` 不能随意用简化固定值替代

### 3. 打通第一版搜索链路

已经验证：

- 小批量请求可稳定通过
- 至少连续 10 条搜索请求未出现新的 `400/403/429`
- 当前第一版“搜索摘要抓取 -> 入库”链路可用

当前结果表明：

- 知乎搜索接口在低频、小批量条件下已具备实用性
- 第四阶段可以先基于搜索结果开展企业增强

### 4. 已加入错误记录

当前已实现：

- 知乎非 200 响应自动写入 `debug/`
- 终端输出对应 debug 文件路径
- `jobs_cleaned.company_check_status` 在失败时写为 `failed`

调试文件示例：

- `debug/zhihu_http_400_*.txt`

文件内容包括：

- 请求方法
- 请求 URL
- 请求头
- 响应状态码
- 响应头
- 响应体

### 5. 已加入搜索结果筛选逻辑

当前只分析搜索结果前 `3` 条。

当前逻辑：

- 读取知乎搜索接口响应 JSON
- 提取前三条的：
  - `title`
  - `excerpt`
- 使用知乎侧关键词做第一轮匹配

当前规则：

1. 摘要命中负面词 -> 直接保留
2. 标题命中负面词，摘要没命中 -> 保留
3. 标题命中中立触发词（如“怎么样”“用户评价”），摘要没命中 -> 保留
4. 标题和摘要都不命中 -> 跳过

### 6. 当前有效输出已简化为单一证据数组

当前阶段的核心输出不再依赖多字段评分，而是一个证据数组。

当前约定：

- `company_enriched.negative_sentences_json`
  - 作为当前阶段唯一有效证据输出
  - 内容不是简单字符串列表，而是证据对象数组

每条证据当前包含：

- `schema_version`
- `evidence_id`
- `content_type`
- `content_id`
- `url`
- `title`
- `description`
- `match_type`
- `matched_in`
- `matched_keyword`
- `reason`

当前其余字段暂时保留，但不作为本阶段核心输出：

- `search_results_json`
- `risk_level`
- `risk_reasons_json`

## 本阶段踩过的关键问题

### 1. 不是没爬到，是初版 JSON 解析错了

一开始出现“请求成功但 `evidence=0`”的情况，后续排查确认：

- 知乎返回的结果项很多是：
  - `item.type = "search_result"`
  - 真正内容类型在 `item.object.type`

初版代码错误地把 `item.type` 当成真实内容类型，导致：

- 实际爬到了 JSON
- 但在本地归一化时被错误过滤

修正后，搜索结果才能正常进入匹配逻辑。

### 2. 页面可见内容不等于搜索接口摘要

当前实现读取的是：

- 搜索接口返回 JSON 中的 `title` / `excerpt`

当前没有读取：

- 页面渲染后的完整 DOM
- `.RichContent-inner`
- `data-first-child`
- “阅读全文”展开后的完整内容

这意味着：

- 浏览器页面里能看到的内容
- 不一定全部存在于 API 返回的 `excerpt` 中

因此后续若出现“页面里有，当前证据里没有”，并不一定是抓取失败，更可能是：

- 关键词只出现在摘要截断之后
- 或只出现在展开全文中

## 关于“阅读全文”的结论

本阶段已经专门评估过是否继续做“阅读全文”。

当前结论是：

- **放弃继续做“阅读全文”**

原因如下：

1. 现有搜索摘要链路已经能支撑第一轮企业筛查
2. “阅读全文”不是简单展开字符串，后续大概率需要额外请求或页面交互
3. 不同内容类型的展开方式、请求结构和返回内容并不统一
4. 为了拿全文继续投入，当前性价比不高
5. 相比之下，继续打磨标题/摘要证据提取更划算

因此本阶段明确：

- `need_expand_fulltext = 1` 只保留为历史调试标记，不再作为后续主要实现方向
- 当前版本不再继续推进全文抓取实现
- 后续企业增强主方向改为：**标题/摘要 + 关键词主题化 + 语义匹配补漏**

## 当前阶段性判断

截至本阶段结束，可以确认：

- 第四阶段第一版搜索摘要抓取已经跑通
- 小批量搜索请求当前稳定
- 企业增强可先基于标题和摘要工作
- 单一证据数组输出比复杂评分更适合当前阶段
- “阅读全文”暂不做是合理决策，不属于缺口

## 后续建议

如果后续继续推进第四阶段，优先级建议如下：

1. 将当前知乎关键词库重构为“风险主题 -> 表达变体”
2. 优化标题/摘要证据片段截取规则
3. 在规则匹配之外，引入语义匹配做补漏
4. 观察更多公司样本，校正误判和漏判
5. 等搜索摘要 + 语义匹配方案稳定后，再决定是否需要接评分阶段

当前不建议优先投入：

1. “阅读全文”页面点击实现
2. 正文全文请求适配
3. 更重的 DOM 级抓取方案

## 后续主要方向

当前已经确认，第四阶段后续的主要演进方向不是正文抓取，而是：

1. 保持知乎搜索接口摘要抓取不变
2. 继续只使用标题和摘要作为输入
3. 放弃“阅读全文”方案
4. 用两层识别替代全文抓取：
   - 第一层：关键词/规则匹配
   - 第二层：语义匹配补漏

这样做的原因是：

- 全文抓取投入高，且不同内容类型请求不统一
- 搜索摘要已经能覆盖相当一部分有效信号
- 当前更大的问题是“匹配效果差”，而不是“拿不到更多字”
- 与其继续追全文，不如提升标题/摘要的识别质量

## 在做语义匹配之前要先完成的事

在进入语义匹配之前，当前还有一个前置工作必须先做好：

- **证据截断收敛**

当前问题：

- 命中后截出来的内容容易是一大段
- 对后续 AI 来说噪声偏多
- 实际上当前阶段真正需要的不是整坨文本，而是结构化、短证据

因此下一步在语义匹配之前，应先把输出结构收敛为：

```json
{
  "title": "...",
  "description": "..."
}
```

说明：

- `title`
  - 直接保存命中的搜索结果标题
- `description`
  - 保存截断后的摘要证据片段
  - 长度应尽量短，只保留对判断有用的局部信息

这样做的目的：

1. 先解决“证据太长、噪声太多”的问题
2. 让后续 AI 或语义模块直接消费统一 JSON
3. 避免后面再做一轮格式清洗

也就是说，后续执行顺序应调整为：

1. 优化标题/摘要证据截断
2. 将当前证据输出收敛成 `title + description` JSON
3. 再在这个结构化输出基础上做语义匹配
