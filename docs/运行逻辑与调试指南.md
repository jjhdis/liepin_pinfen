# 运行逻辑与调试指南

这份文档说明当前项目在 `list` / `detail` 两种模式下的运行逻辑，以及遇到风控、解析失败、数据未落库时应该怎么排查。

## 1. 程序入口

入口文件是 [main.py](D:/channel/PythonProject2/main.py)。

命令分为两种：

```powershell
.\.venv\Scripts\python.exe .\main.py list --keyword python --page 0 --pages 1 --interactive
.\.venv\Scripts\python.exe .\main.py detail --keyword python --max-detail 2 --interactive --confirm-every 1 --min-delay 90 --max-delay 180
```

整体流程：

1. `parse_args()` 解析命令行参数。
2. `async_main()` 初始化数据库。
3. 根据 `args.command` 进入 `run_list_mode()` 或 `run_detail_mode()`。

## 2. 模块分工

- [config.py](D:/channel/PythonProject2/config.py)
  负责全局配置，例如城市、默认关键词、浏览器参数、文件路径。
- [crawler/browser.py](D:/channel/PythonProject2/crawler/browser.py)
  负责启动 Playwright、创建浏览器上下文、注入 cookie、注入基础 stealth 脚本。
- [crawler/anti_detect.py](D:/channel/PythonProject2/crawler/anti_detect.py)
  负责随机等待、鼠标移动、页面滚动。
- [crawler/list_page.py](D:/channel/PythonProject2/crawler/list_page.py)
  负责打开搜索页、模拟浏览、提取 `job_id`。
- [crawler/detail_page.py](D:/channel/PythonProject2/crawler/detail_page.py)
  负责打开详情页、解析 `ld+json`、提取结构化职位数据。
- [storage/database.py](D:/channel/PythonProject2/storage/database.py)
  负责 `jobs` 和 `crawl_log` 两张表的读写。

## 3. 浏览器初始化逻辑

`open_browser()` 做的事情：

1. 启动 Chromium。
2. 创建 browser context。
3. 设置 `user_agent`、`locale=zh-CN`、`timezone_id=Asia/Shanghai`、固定 viewport。
4. 通过 `add_init_script()` 注入基础 stealth 脚本。
5. 从项目根目录读取 `cookies.json` 并注入到 context。

这意味着：

- 只要 `cookies.json` 是当前账号有效 cookie，后面新开的 page 都会继承这份登录态。
- 如果 `cookies.json` 过期，最常见现象是被重定向到登录页，或者详情页被安全页替换。

## 4. `list` 模式运行逻辑

`run_list_mode()` 的执行顺序：

1. 调用 `open_browser()` 创建浏览器上下文。
2. `context.new_page()` 创建页面。
3. 对每个页码调用 `scrape_list_page(page, keyword, page_no)`。
4. 抓到 `job_id` 后，把 stub 数据写入 `jobs` 表。

`scrape_list_page()` 的内部逻辑：

1. 根据关键词和页码拼出搜索 URL。
2. `page.goto(..., wait_until="networkidle")` 打开列表页。
3. 调用 `human_like_scroll(page)` 做鼠标移动和滚动。
4. 优先从 DOM 里的 `a[href*='/a/']` 提取 `job_id`。
5. 如果 DOM 没抓到，再回退到正则扫整页 HTML。
6. 如果还是没有，就把 HTML 写到 `debug/list_{keyword}_{page}.html`。

列表页风控判断：

- 页面标题包含“验证码”或“安全中心”。
- 页面跳到了登录页。

## 5. `detail` 模式运行逻辑

`run_detail_mode()` 先从数据库读取待补全的职位：

```sql
SELECT job_id, keyword, detail_url
FROM jobs
WHERE title IS NULL OR TRIM(title) = ''
ORDER BY created_at ASC
LIMIT ?
```

也就是说：

- `list` 模式只是把职位骨架写进去。
- `detail` 模式才真正补全标题、薪资、JD、公司等字段。

当前 `detail` 的执行顺序是：

1. 读取待抓详情职位。
2. 打开一个浏览器 page。
3. 每处理一条职位前，先执行 `open_search_page(page, keyword, 0)`。
4. 搜索页加载后会先做一轮滚动。
5. 再执行 `fetch_detail_page(page, job_id, keyword, referer=page.url)`。
6. 详情页打开后做少量鼠标移动和滚动。
7. 读取 HTML，调用 `parse_detail_page()` 解析。
8. `upsert_job()` 写回 `jobs` 表。
9. `log_crawl()` 把本次抓取结果记录到 `crawl_log`。
10. 如果不是最后一条，则执行 `random_delay(min_delay, max_delay)`。

这里最关键的变化是第 3 步和第 5 步：

- 以前是直接连续 `goto(详情页 URL)`。
- 现在是先进入搜索页，再带 `referer` 访问详情页。

这让访问路径更接近真实用户，因此你这次用：

```powershell
.\.venv\Scripts\python.exe .\main.py detail --keyword python --max-detail 2 --interactive --confirm-every 1 --min-delay 90 --max-delay 180
```

没有触发风控。

## 6. 详情页解析逻辑

`parse_detail_page()` 的核心步骤：

1. 用 BeautifulSoup 解析 HTML。
2. 如果标题里有“验证码”或“安全中心”，直接抛 `DetailPageBlockedError`。
3. 扫描所有 `script[type="application/ld+json"]`。
4. 优先找 `@type == "JobPosting"` 的结构化数据。
5. 提取标题、薪资、城市、经验、学历、发布时间、公司信息、JD、福利。
6. 补充 `.time-factor-wrap` 中的更新时间。
7. 计算 `days_since_update`。
8. 如果最后标题仍然为空，抛异常。

因此详情抓取失败通常分三类：

- 被风控页替换：抛 `DetailPageBlockedError`
- 页面结构变了：能打开，但 `title` 等关键字段没解析出来
- 登录态失效：页面被跳转，最终解析不到职位内容

## 7. 数据库怎么判断程序跑到了哪一步

数据库文件是 [jobs.db](D:/channel/PythonProject2/jobs.db)。

### 7.1 看列表页是否写入 stub

如果 `list` 成功，`jobs` 表里会出现：

- `job_id`
- `keyword`
- `detail_url`
- `title = NULL`

这表示列表页抓到了，但详情还没补全。

### 7.2 看详情页是否补全成功

如果 `detail` 成功，`jobs` 表里会出现：

- `title`
- `salary_min`
- `salary_max`
- `company_name`
- `jd_text`
- `raw_html`
- `raw_json`

### 7.3 看 `crawl_log`

`crawl_log` 用来判断每次请求是成功、失败还是被风控。

重点字段：

- `url`
- `job_id`
- `success`
- `error_message`
- `latency_seconds`
- `created_at`

你可以直接查最近日志：

```powershell
sqlite3 .\jobs.db "select id, url, job_id, success, error_message, created_at from crawl_log order by id desc limit 20;"
```

如果你本机没有 `sqlite3` 命令，也可以后面让我给你加一个 Python 查询脚本。

## 8. 调试时优先看什么

### 8.1 列表页没抓到数据

先看：

- 控制台是否打印 `[list-blocked]`
- `debug/` 目录是否生成 HTML
- `jobs` 表里是否有新 stub

如果没有 `job_id`，常见原因是：

- 页面还没完全加载
- DOM 结构变了
- 实际返回的是风控页或登录页

### 8.2 详情页第二条开始风控

先看：

- 当前命令的 `--min-delay` / `--max-delay`
- 是否还在直连详情页
- cookie 是否是刚更新的
- `crawl_log.error_message` 是否是 `detail page was replaced by captcha`

当前版本已经把“先搜索页预热，再进详情”做进去了。如果以后又出现第二条就风控，优先怀疑：

- cookie 过期
- 新账号信誉更低
- 详情抓取频率又调快了
- 猎聘开始校验更强的访问链路

### 8.3 详情页打开了但解析失败

看：

- 控制台是否打印 `[detail-failed]`
- `error_message` 是不是 `detail page parsed but title is still missing`
- `raw_html` 里实际返回的页面是不是职位页

这种情况通常不是风控，而是页面结构变化或解析规则不够稳。

## 9. 推荐的调试顺序

以后遇到问题，按这个顺序查：

1. 先确认 `cookies.json` 是当前账号、当前会话导出的。
2. 跑 `list` 模式，确认能稳定写入 stub。
3. 跑 `detail` 模式，但 `--max-detail 1` 先只抓一条。
4. 再把 `--max-detail` 提到 `2`，同时拉大 `--min-delay` / `--max-delay`。
5. 检查 `crawl_log` 的 `error_message`。
6. 如果解析失败，再看 `jobs.raw_html` 或 `debug/` 里的 HTML。

## 10. 目前最稳的运行建议

列表页：

```powershell
.\.venv\Scripts\python.exe .\main.py list --keyword python --page 0 --pages 1 --interactive
```

详情页：

```powershell
.\.venv\Scripts\python.exe .\main.py detail --keyword python --max-detail 2 --interactive --confirm-every 1 --min-delay 90 --max-delay 180
```

如果这个配置稳定，再慢慢把 `max-detail` 提高，不要一次把间隔降得太快。

## 11. 下一步如果还要增强稳定性

如果后面仍然偶发详情风控，下一层优化方向是：

1. 不再 `goto(详情 URL)`，改成从列表页真实点击进入详情。
2. 给详情页之间增加更长的阅读停顿。
3. 记录每次被风控时的 HTML 到 `debug/`。
4. 把 `crawl_log` 扩展成记录页面标题、最终 URL、是否命中验证码页。

这几项都比当前版本更重，但如果站点继续收紧策略，会有价值。
