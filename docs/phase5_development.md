# 第五阶段开发文档

## 目标

第五阶段聚焦“稳定性强化与批量运行”。

这一阶段的核心目标不是新增更多花哨能力，而是把现有链路在更高数据量下跑稳，并为后续前端管理页和多招聘站扩展打基础。

当前阶段的目标：

- 支撑更大规模的抓取与后处理联调
- 持续修复抓取、清洗、企业增强、评分中的异常分支
- 建立清晰的失败分类、重试边界和人工介入点
- 为后续前端页面提供稳定的状态模型、日志输出和结果查询结构

## 当前背景

截至第五阶段开始前，系统已经具备以下链路：

1. 第一阶段抓取 -> `jobs`
2. 第二阶段清洗 -> `jobs_cleaned`
3. 第四阶段企业增强 -> `company_enriched`
4. 第三阶段评分 -> `scores`

同时，已经补充：

- 知乎增强改为前 `10` 条抓取 + filter 后保留少量证据
- `company_enriched` 已保存：
  - `zhihu_raw_results_json`
  - `zhihu_filtered_results_json`
- 评分阶段已能将 `company_research` 拼进模型输入
- 已新增后处理串联入口：
  - `postprocess_pipeline.py`

这意味着当前系统已经进入“可以开始批量联调”的阶段。

## 当前范围

第五阶段已明确纳入：

- 猎聘详情抓取稳定性修复
- 大批量样本联调与问题收敛
- 后处理链路：
  - `clean_jobs.py`
  - `company_enrich.py`
  - `score_jobs.py`
  - `postprocess_pipeline.py`
- 失败类型分类、日志与调试产物收敛
- 为前端管理页预留状态字段、统计逻辑和查询结构

当前不纳入：

- 前端页面正式实现
- 多招聘站抓取正式接入
- 分布式任务调度
- 重型工作流系统（如 Airflow / Celery）

## 设计原则

- 抓取与后处理继续解耦
- 各阶段仍保留独立入口，不重写已有业务逻辑
- 优先做“明确分类 + 能重跑 + 可观察”，而不是过早追求自动化架构
- 对不可恢复样本与可重试样本做清晰区分
- 所有新增状态与日志要为后续前端直接复用

## 第五阶段主任务

### 0. 半自动登录与 Cookie 刷新工具

这是第五阶段开头优先级很高的一项。

原因：

- 当前猎聘与知乎链路都依赖登录态
- 批量运行时，Cookie 失效会直接影响抓取与企业增强
- 全自动登录长期不稳定，但半自动登录非常适合当前阶段

当前建议方案：

- 提供半自动登录辅助脚本
- 自动打开登录页
- 自动跳转到正确入口
- 自动填写手机号或预留输入位置
- 自动点击发送验证码
- 登录成功后自动导出并保存 Cookie

明确边界：

- 不追求自动读取短信验证码
- 不追求自动绕过滑块 / 风控
- 允许人工完成验证码、滑块和确认步骤
- 成功后由脚本负责落库 / 落文件保存 Cookie

当前阶段优先目标：

- 至少支持猎聘 Cookie 半自动刷新
- 后续再补知乎 Cookie 半自动刷新
- 统一形成“Cookie 失效 -> 人工登录 -> 自动导出 Cookie -> 恢复运行”的闭环

### 1. 大规模抓取联调

目标：

- 支撑单日约 `200` 条量级的持续联调
- 暴露高频异常并逐步收敛

当前策略：

- 继续保持低频抓取
- 批量运行后处理链路
- 遇到新异常先分类，再决定是否修复、跳过、删除或重试

### 2. 详情页稳定性强化

当前已识别的典型异常包括：

- `blocked`
  - 验证码 / 安全中心 / 风控页
- `login_required`
  - 被跳转到登录拦截页
- `expired`
  - 职位过期 / 职位失效页
- `wrong_page`
  - 落到非目标详情页
- `parse_failed`
  - 页面可打开，但解析失败

当前已明确：

- `blocked` / `login_required`
  - 终止当前批次，等待人工处理
- `expired` / `wrong_page`
  - 记录失败并删除对应职位数据，避免反复进入 pending
- `parse_failed`
  - 保留调试文件，继续分析原因

### 3. 后处理链路稳定运行

当前已增加：

- `postprocess_pipeline.py`

职责：

1. 调用 `clean_jobs.py`
2. 调用 `company_enrich.py`
3. 调用 `score_jobs.py`

设计边界：

- 不重写清洗、增强、评分逻辑
- 只做顺序调用和最小编排
- 作为后续前端“后处理按钮”的统一入口

### 4. 企业增强与评分联调

当前重点不是继续扩展知乎抓取方式，而是验证：

- `zhihu_filtered_results_json` 是否足够干净
- `company_research` 是否足够支持 `score_company`
- `score_company` 是否会被噪声证据误导

当前评分输入结构已经扩展为：

- 职位基础信息
- `company_research.risk_level`
- `company_research.evidence`

第五阶段需要持续观察：

- 证据为空时的评分表现
- 弱相关证据是否导致误扣分
- 泛行业讨论是否仍会误入评分输入

## 必须完成的事项

以下事项属于第五阶段必须完成内容：

### 1. 明确抓取状态模型

当前系统仍有部分逻辑通过“标题是否为空”间接推断状态。

第五阶段应逐步收敛到显式状态，例如：

- `pending`
- `success`
- `expired`
- `wrong_page`
- `login_required`
- `blocked`
- `parse_failed`

目的：

- 提高可观察性
- 减少后续前端查询时的隐式判断

### 2. 建立失败分类统计

必须能按批次或按天统计：

- 成功多少
- 过期多少
- 错页多少
- 登录拦截多少
- 风控多少
- 解析失败多少

这是后续优化抓取质量的基础。

### 3. 区分可重试与不可重试

建议边界：

- 不可重试：
  - `expired`
  - `wrong_page`
- 可重试但需人工介入或等待：
  - `login_required`
  - `blocked`
  - 网络异常
- 待继续分析：
  - `parse_failed`

### 4. 固化结果查询结构

在前端实现前，必须先明确最终结果查询结构。

后续前端需要至少能组合查询：

- `jobs_cleaned`
- `company_enriched`
- `scores`

例如：

- 职位基本信息
- 公司增强证据
- AI 评分结果
- 投递建议

### 5. 固化 Cookie / 登录维护流程

第五阶段必须形成明确约定：

- Cookie 文件位置
- Cookie 失效后的日志特征
- 人工更新流程
- 更新后的恢复验证方式

否则批量运行时会反复中断。

同时，第五阶段建议配套实现：

- 半自动登录刷新 Cookie 工具
- Cookie 是否失效的显式日志提示
- 更新 Cookie 后的快速验证脚本

## 第五阶段建议新增能力

### 1. 更明确的运行日志

建议日志统一输出：

- 阶段名
- job_id / company_name
- 状态
- 原因
- 后续动作

例如：

- `detail-blocked`
- `detail-skip-delete`
- `company-enrich-stop`
- `score-summary`

### 2. 更明确的批次摘要

建议每次运行后输出：

- 本批总数
- 成功数
- 跳过数
- 删除数
- 失败数

这部分日志后续可直接被前端消费。

### 3. 前端状态字段预留

后续前端会关心：

- 当前抓取是否运行
- 当天新增职位数
- 待清洗数量
- 待企业增强数量
- 待评分数量
- 最近错误类型

第五阶段至少要让这些统计在后端可查询。

## 与前端阶段的衔接

第五阶段不是直接做前端，但必须为前端提供稳定基础。

后续前端预计包含两类功能：

### 1. 爬虫与流程管理

- 启动 / 停止抓取
- 查看当前批次状态
- 查看累计抓取数量
- 达到阈值后触发后处理链路

### 2. 数据与评分结果查看

- 查看清洗后职位
- 查看企业增强证据
- 查看 AI 评分结果
- 最终按 SQL 或视图组装输出

因此第五阶段的重点是：

- 先把状态、日志、查询结构定清楚
- 再进入前端实现

## 当前阶段性结论

第五阶段的本质不是“继续堆功能”，而是：

- 让系统在更大规模样本下稳定
- 让失败可分类、可观察、可处理
- 让后处理链路形成统一入口
- 为前端和多招聘站扩展打基础

## 下一步建议

第五阶段开始后，建议按以下顺序推进：

1. 大规模抓取联调，持续收敛异常
2. 补齐失败分类统计与状态模型
3. 稳定运行 `postprocess_pipeline.py`
4. 明确最终查询 SQL / 视图结构
5. 再开始前端页面开发
