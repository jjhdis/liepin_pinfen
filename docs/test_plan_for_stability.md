## 后续测试方案：验证单 Cookie 稳定阈值与策略有效性

> 基于当前已实现的策略（主动停止、异常职位跳过、随机延时、单次运行时长限制等），需要系统性地回答以下问题：
> - 30 条是稳定均值还是偶发上限？
> - 是否存在 20 条就触发、或偶尔能到 40 条的情况？
> - 关键词、时间段、Cookie 新鲜度是否影响阈值？
> - 冷却后二轮复用是否可行？

### 一、测试前提

- 使用当前已落地的策略配置（不中途修改代码）
- 只改变少量输入变量（关键词、时间、Cookie）
- 每轮测试独立记录关键数据

### 二、基础测试：固定条件跑 5 轮

**固定变量：**
- 同一个 Cookie（未触发过手机号验证）
- 同一个关键词（如 `python`）
- 同一时间段（如每天上午 10 点）
- 使用现有延时、主动停、异常跳过逻辑

**每轮记录：**

| 轮次 | 成功条数 | 是否触发验证码 | 触发时累计条数 | 失败原因（如有） |
|------|----------|----------------|----------------|------------------|
| 1    |          |                |                |                  |
| 2    |          |                |                |                  |
| 3    |          |                |                |                  |
| 4    |          |                |                |                  |
| 5    |          |                |                |                  |

**结论判断：**
- 若 5 轮中 4 轮在无验证码前提下跑到 ≥25 条 → 当前策略稳定
- 若多轮在 20 条以内触发 → 阈值需下调至 20

### 三、单变量对照测试（每项跑 3 轮）

一次只改变一个变量，与基础测试结果对比。

#### 变量 A：不同关键词
- 换为 `java` / `测试` / `产品经理`
- 验证阈值是否与关键词热度或列表页质量相关

#### 变量 B：不同时间段
- 晚间 9 点
- 凌晨 / 清晨
- 观察风控强度是否随时间变化

#### 变量 C：不同 Cookie
- 用另一个账号的新登录态（未使用过）
- 判断 Cookie 老化是否影响阈值

### 四、冷却复用验证（最关键）

**目的：** 验证“主动停止 + 冷却数小时后当天是否还能跑第二轮”

**步骤：**
1. 选一个未触发过验证码的 Cookie
2. 第一轮跑到 20 条主动停止
3. 冷却 4–6 小时
4. 第二轮继续抓取，记录成功条数及是否触发验证

**判断标准：**
- 第二轮成功 ≥15 条且无验证 → 冷却复用可行，总量可提升
- 第二轮很快触发验证 → 单 Cookie 当天只适合一轮

### 五、测试陷阱提醒

- **不把单次 40 条当成常态**：看多轮中位数，而非最大值
- **测试期间不改策略**：保证数据可比
- **区分失败类型**：手机号验证 vs 滑块验证 vs 无结果页

### 六、稳定性通过标准

> **连续 5 轮中，至少有 4 轮在不触发任何验证码的前提下跑到 ≥25 条**

达到此标准：
- 当前策略对当前环境有效
- 可将此作为基线，再尝试提高条数或引入冷却复用

未达到标准：
- 实际安全边界更低（如 20 条）
- 需检查策略是否有漏洞，或必须走向多 Cookie 方案

完成上述测试后，将结果汇总至 `phase5_issue_log.md` 的“测试结论”小节，作为是否进入 Phase6（Cookie 池管理）的依据。
