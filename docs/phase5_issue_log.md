# Phase5 问题记录

## 2026-06-02 `detail-failed` 日期格式异常

### 现象

抓取详情页时出现：

```text
[detail-failed] job_id=76712787 error=Invalid isoformat string: '2026-12-31T23:59'
```

### 原因

这不是风控，也不是页面跳错。

根因是详情页解析阶段把 `last_updated` 当成了“纯日期”处理，但这条职位返回的是带时间的字符串：

```text
2026-12-31T23:59
```

当前详情解析逻辑里会执行：

```python
date.fromisoformat(job["last_updated"])
```

这里 `date.fromisoformat()` 期望的是：

```text
YYYY-MM-DD
```

但实际拿到的是：

```text
YYYY-MM-DDTHH:MM
```

所以直接抛出 `Invalid isoformat string`。

### 影响

- 当前这条职位详情抓取失败
- 失败类型属于“解析逻辑格式兼容不足”
- 不属于 Cookie 失效
- 不属于验证码/风控
- 不属于错页或职位过期

### 当前判断

这类问题本质上是日期归一化缺口：

- `ld+json` 里有些字段返回纯日期
- 也有些字段返回完整 datetime
- 当前代码只兼容了纯日期场景

### 后续修复建议

后续等你这轮爬取结束后再处理，建议修复方向：

1. 先把 `last_updated` / `date_posted` 统一归一化成 `YYYY-MM-DD`
2. 或在计算 `days_since_update` 时兼容 datetime，再取日期部分
3. 同类字段都要统一处理，避免只修一处

### 当前处理策略

本轮先记录，不改爬虫代码，避免影响你正在跑的抓取任务。

## 2026-06-02 已关闭 / 异常职位重复访问触发风控

### 现象

在 `detail` 模式中，前面一批职位访问正常；当某条职位实际已关闭、不存在或返回异常页后，短时间内连续重复访问同一目标，随后在“搜索页预热”阶段出现：

```text
[detail-failed] job_id=... error=list page was replaced by captcha
```

### 当前判断

这类问题大概率不是单次访问关闭职位立即触发风控，而是以下组合导致风险升高：

- 同一条失效职位被短时间重复访问
- 访问不存在/异常详情页
- 失败后再次走 `搜索页预热 -> 详情页` 的完整链路

也就是说，真正危险的不是“关闭职位”本身，而是：

- 已知异常目标被连续重复打
- 同一组高风险访问路径在短时间内重复执行

### 对后续多站点的通用意义

这个坑后面接别的招聘站也大概率会重复出现，只是触发形式不同：

- 失效职位
- 下线职位
- 错误详情页
- 跳转页
- 反爬验证页

共性都是：

- 异常目标第一次访问不可完全避免
- 但同一异常目标不应被主链路连续重复访问

### 当前结论

Phase5 的目标不是完全不踩坑，而是让同类坑只踩一次。

后续应尽量满足：

1. 失效/异常职位命中后尽快从原始抓取队列剔除
2. 同一异常目标不要在短时间内再次进入 `detail`
3. 一旦出现 `captcha/login_required`，当前批次应立即停止
4. 这类风控坑应当作为“多站点通用风险模式”保留

## 2026-06-03 新结论：`expired / wrong_page` 不是普通 skip，而是停批信号

### 新观察

本轮详情抓取中，前面职位连续成功；在命中一条：

- `detail_status = expired`

之后，虽然程序原先按 `skip` 逻辑继续运行，但仅继续处理约 `2` 条后，就出现：

```text
[detail-stop] job_id=... status=blocked error=list page was replaced by captcha
```

本轮关键片段：

```text
[detail-success] ...
[detail-success] ...
[detail-skip] job_id=66232597 status=expired error=detail page is expired or no longer available
[detail-cooldown] processed=20 ...
[detail-success] ...
[detail-stop] job_id=75626487 status=blocked error=list page was replaced by captcha
```

### 当前判断

这说明在当前阶段，`expired`（以及同类的 `wrong_page`）不能再被理解成普通业务异常后“跳过继续”。

更合理的理解是：

1. 命中这类异常详情页，本身就会明显抬高当前会话风险
2. 即使后面紧接着还能成功 1-2 条，也不代表风险已经恢复
3. 继续跑当前批次，极容易在随后的搜索页预热或详情访问中触发验证码

也就是说：

- `expired / wrong_page`
  - 在当前阶段应视为 **风控前兆**
  - 而不是普通 `skip`

### 已调整策略

当前代码策略已同步改为：

1. 命中 `expired / wrong_page`
2. 记录状态
3. 删除对应原始 job
4. **立即停止当前批次**

不再允许：

- `expired / wrong_page -> skip -> 继续抓后续 detail`

### 当前阶段结论

截至目前，详情抓取主链路的停批条件应理解为三类：

1. `blocked`
2. `login_required`
3. `expired / wrong_page`

其中前两类属于显式风控；第三类属于**高风险前兆，需按停批处理**。

## 2026-06-03 当前最保守运行基线

### 当前策略

结合近几轮实测，当前先按最保守方案运行：

1. 单个 Cookie
2. 单个 keyword
3. 单轮最多抓 `20` 条 detail
4. 单个列表页默认只保留前 `20` 条职位进入待抓详情队列

### 当前判断

这套策略的目的不是追求单轮极限，而是先保证：

- 尽量不碰列表尾部更高风险职位
- 尽量不接近单 Cookie 的高风险区间
- 让多 Cookie 轮换更稳定

当前阶段不再把以下目标作为默认策略：

- 单 Cookie 常规跑 `25-30` 条
- 单页保留 `30-40` 条列表尾部职位
- 在同一轮里为了提量继续放宽边界

### 当前配置约定

当前已将默认保守值同步进配置层：

- `RUN_CONFIG["list"]["store_top_n"] = 20`

也就是说，后续如果不在命令行显式覆盖，`list` 模式会默认只保留每页前 `20` 条。

## 2026-06-03 新结论：踩到地雷后即使立刻停，也可能仍会触发延迟手机号验证

### 新观察

本轮已经确认一种更糟的情况：

1. 详情抓取过程中命中地雷职位
2. 程序已按新策略立即停止当前批次
3. 但中午离开一段时间后回来看，当前猎聘会话仍然进入了手机号验证状态

也就是说：

- 不是只有“踩雷后继续跑很多条”才会触发验证
- 也存在“踩雷那一刻就已经被判定高风险，只是验证稍后才显示出来”的情况

### 当前判断

这说明当前阶段应把“地雷职位”理解为：

- **高危触发点**

而不只是：

- “普通异常职位”
- “停批后还能保住当前 Cookie 的风险点”

当前更准确的理解应为：

1. 停批仍然是必须的
2. 但停批只能防止继续扩大损害
3. 停批不能保证当前 Cookie 一定幸免
4. 真正关键的是尽量不要踩到地雷本身

### 当前阶段结论

截至目前，稳定性的核心矛盾已经进一步收敛为：

- **如果 list 端排雷不彻底，整个链路就很难真正稳定**

因为一旦踩到地雷：

- 当前 Cookie 很可能已经进入待验证状态
- 即使程序已经立刻停掉，也不代表这次会话能保住

因此后续“隐藏卡片过滤 / 地雷排除”是否有效，将直接决定：

- 单 Cookie 上限能否回到更高区间
- 整体链路是否具备稳定运行可能

## 2026-06-02 当前实测阈值：约 30 条后触发手机号验证

### 当前观察

目前已经确认：

- 在数据抓取正常、详情抓取连续成功的情况下
- 仍然会在累计约 `30` 条后触发手机号验证

这说明当前风险并不只来自异常页或关闭职位，正常成功抓取本身也存在累计阈值。

### 当前判断

这个 `30` 条是当前阶段的实测观察值，不先视为绝对固定规则，但足以作为 Phase5 的临时参考阈值。

### 当前阶段结论

在现有节奏下，单个 Cookie 连续跑到约 `30` 条已经接近高风险边界。

因此当前阶段至少要接受以下结论：

1. 单 Cookie 不适合长批次连续跑
2. 跑到约 `30` 条时，需要长时间休息，或切换 Cookie
3. 即使前面全是成功样本，也不能认为风险在下降

### 当前处理方式

本阶段先只记录观察结论，不直接写死自动切换逻辑。

后续等更多样本验证后，再决定：

- 这个阈值是按“条数”生效
- 还是按“时间 + 条数”共同生效
- 以及长休息多长时间才足够安全

## 2026-06-02 当前阶段结论：同一 Cookie 触发手机号验证后，当天不可继续使用

### 当前观察

已确认以下实际情况：

- 早上约 `10` 点触发过风控手机号验证
- 随后重新获取同一账号/同一 Cookie 的新登录态
- 当天再次尝试继续爬取，仍然无法恢复正常抓取

### 当前结论

当前阶段可以先按以下规则理解：

1. 同一 Cookie / 同一账号一旦触发手机号验证，不是“刷新 Cookie 后立刻恢复”
2. 即使重新获取了最新 Cookie，当天继续使用仍然大概率不可行
3. 这类 Cookie 至少应按“冷却一天”处理

### 当前影响

这说明当前系统的恢复方式不能理解成：

- 触发验证
- 重新导出 Cookie
- 立刻继续跑

而应理解成：

- 一旦触发手机号验证
- 当前 Cookie / 账号当天退出生产使用
- 至少进入 `1` 天级别冷却

### 当前阶段处理方式

本阶段先将其记录为实测规则，不立即自动化实现。

后续 Phase6 设计 Cookie profile / 账号池时，这条应视为高优先级约束：

- 触发手机号验证的 Cookie profile
- 不能在当天再次投入抓取

## 2026-06-02 当前阶段目标修正：从 200 条下调到 120 条

### 当前背景

结合目前已知观察：

- 单 Cookie 正常连续抓取约 `30` 条接近风控边界
- 异常职位会明显抬高风险
- 一旦触发手机号验证，当天基本不可恢复

原先按单日 `200` 条设计，当前阶段现实性偏低。

### 当前更现实的目标

当前阶段更合理的目标应调整为：

- 找到 `3` 个可用 Cookie
- 在不触发手机号验证的前提下轮换使用
- 总量先以约 `120` 条为目标

可理解为：

- 单 Cookie 目标量级约 `30-40` 条
- `3` 个 Cookie 合计目标约 `120` 条

### 当前结论

现阶段不应再把 `200` 条作为默认可达目标，而应先按：

- `120` 条可稳定运行
- 且不触发手机号验证

作为当前更现实的阶段性目标。

### 当前阶段意义

这条目标修正意味着：

1. 后续策略设计要围绕“低风险稳定 120 条”展开
2. `200` 条不作为当前阶段默认基线
3. 先把“多 Cookie 低风险轮换”跑通，再评估是否有必要继续往上推量

## 2026-06-02 最佳情况目标：同一 Cookie 当天冷却后可二轮复用

### 目标设想

当前最理想的运行模型不是：

- 每个 Cookie 当天只跑一次

而是：

- 单个 Cookie 在**未触发手机号验证**的前提下
- 首轮跑到安全边界前主动停止
- 冷却若干小时后
- 当天还能再次投入第二轮抓取

### 如果该假设成立

则后续更理想的总量模型可能变成：

- 前三轮先各跑约 `30`
- 合计约 `90`
- 再让最早停下的 Cookie 经过长冷却后回到生产
- 额外补跑约 `10-20`

这样总量就不再被严格锁死在：

- `4 个 Cookie * 30 = 120`

而可能允许：

- `3-4` 个 Cookie
- 通过冷却后二轮复用
- 把总量进一步往上推

### 当前状态

这还只是当前阶段的**最佳情况目标**，尚未验证。

当前还不能把它当成既定规则使用。

### 后续要验证的问题

后续重点不是“触发手机号验证后能否恢复”，而是：

1. 单个 Cookie 在**未触发验证**的前提下
2. 跑到边界前主动停止
3. 冷却数小时后
4. 当天是否还能稳定跑第二轮

### 当前意义

如果该假设成立，后续 Phase5 / Phase6 的策略空间会明显变大；
如果不成立，则应按“单 Cookie 单天单轮”继续保守设计。

## 2026-06-02 新实验结果：触发风控后冷却约 6 小时存在恢复可能

### 当前观察

新增一条关键实验结果：

- 大约 `12` 点左右，单 Cookie 在累计约 `30` 条后触发风控
- 随后更新了 Cookie
- 冷却约 `6` 小时后再次尝试
- 该 Cookie 能继续爬取

### 对前结论的修正

这说明前面“同一 Cookie 触发手机号验证后，当天不可继续使用”的结论过于绝对。

当前更合理的理解应调整为：

1. 触发风控后**不能立即恢复**
2. 但**更新 Cookie + 长时间冷却后，存在恢复可能**
3. 当前观察到的可用恢复窗口，至少已有一个样本接近 `6` 小时

### 当前更准确的阶段结论

现阶段更保守且更贴近事实的说法应为：

- 单 Cookie 约 `30` 条接近风控边界
- 触发风控后不能立即继续跑
- 更新 Cookie 并冷却约 `6` 小时后，存在再次恢复可用的可能
- 是否总能恢复、恢复后能跑多少条，仍需继续验证

### 当前待继续验证的问题

后续还需要继续确认：

1. `6` 小时是否是稳定有效的恢复窗口
2. 恢复后单 Cookie 还能继续跑多少条
3. 恢复后的风险边界是否仍接近 `30` 条
4. 这条规律是否对不同 Cookie 都成立
