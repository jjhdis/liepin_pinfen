# Phase5 问题记录

## 2026-06-02 `detail-failed` 日期格式异常

### 现象

抓取详情页时出现：

```text
[detail-failed] job_id=76712787 error=Invalid isoformat string: '2026-12-31T23:59'
```

### 原因

这不是风控，也不是页面跳错。

根因是详情页解析阶段把 `last_updated` 当成了“纯日期”处理，但这条职位返回的是带时间的字符串：

```text
2026-12-31T23:59
```

当前详情解析逻辑里会执行：

```python
date.fromisoformat(job["last_updated"])
```

这里 `date.fromisoformat()` 期望的是：

```text
YYYY-MM-DD
```

但实际拿到的是：

```text
YYYY-MM-DDTHH:MM
```

所以直接抛出 `Invalid isoformat string`。

### 影响

- 当前这条职位详情抓取失败
- 失败类型属于“解析逻辑格式兼容不足”
- 不属于 Cookie 失效
- 不属于验证码/风控
- 不属于错页或职位过期

### 当前判断

这类问题本质上是日期归一化缺口：

- `ld+json` 里有些字段返回纯日期
- 也有些字段返回完整 datetime
- 当前代码只兼容了纯日期场景

### 后续修复建议

后续等你这轮爬取结束后再处理，建议修复方向：

1. 先把 `last_updated` / `date_posted` 统一归一化成 `YYYY-MM-DD`
2. 或在计算 `days_since_update` 时兼容 datetime，再取日期部分
3. 同类字段都要统一处理，避免只修一处

### 当前处理策略

本轮先记录，不改爬虫代码，避免影响你正在跑的抓取任务。

## 2026-06-02 已关闭 / 异常职位重复访问触发风控

### 现象

在 `detail` 模式中，前面一批职位访问正常；当某条职位实际已关闭、不存在或返回异常页后，短时间内连续重复访问同一目标，随后在“搜索页预热”阶段出现：

```text
[detail-failed] job_id=... error=list page was replaced by captcha
```

### 当前判断

这类问题大概率不是单次访问关闭职位立即触发风控，而是以下组合导致风险升高：

- 同一条失效职位被短时间重复访问
- 访问不存在/异常详情页
- 失败后再次走 `搜索页预热 -> 详情页` 的完整链路

也就是说，真正危险的不是“关闭职位”本身，而是：

- 已知异常目标被连续重复打
- 同一组高风险访问路径在短时间内重复执行

### 对后续多站点的通用意义

这个坑后面接别的招聘站也大概率会重复出现，只是触发形式不同：

- 失效职位
- 下线职位
- 错误详情页
- 跳转页
- 反爬验证页

共性都是：

- 异常目标第一次访问不可完全避免
- 但同一异常目标不应被主链路连续重复访问

### 当前结论

Phase5 的目标不是完全不踩坑，而是让同类坑只踩一次。

后续应尽量满足：

1. 失效/异常职位命中后尽快从原始抓取队列剔除
2. 同一异常目标不要在短时间内再次进入 `detail`
3. 一旦出现 `captcha/login_required`，当前批次应立即停止
4. 这类风控坑应当作为“多站点通用风险模式”保留

## 2026-06-02 当前实测阈值：约 30 条后触发手机号验证

### 当前观察

目前已经确认：

- 在数据抓取正常、详情抓取连续成功的情况下
- 仍然会在累计约 `30` 条后触发手机号验证

这说明当前风险并不只来自异常页或关闭职位，正常成功抓取本身也存在累计阈值。

### 当前判断

这个 `30` 条是当前阶段的实测观察值，不先视为绝对固定规则，但足以作为 Phase5 的临时参考阈值。

### 当前阶段结论

在现有节奏下，单个 Cookie 连续跑到约 `30` 条已经接近高风险边界。

因此当前阶段至少要接受以下结论：

1. 单 Cookie 不适合长批次连续跑
2. 跑到约 `30` 条时，需要长时间休息，或切换 Cookie
3. 即使前面全是成功样本，也不能认为风险在下降

### 当前处理方式

本阶段先只记录观察结论，不直接写死自动切换逻辑。

后续等更多样本验证后，再决定：

- 这个阈值是按“条数”生效
- 还是按“时间 + 条数”共同生效
- 以及长休息多长时间才足够安全

## 2026-06-02 当前阶段结论：同一 Cookie 触发手机号验证后，当天不可继续使用

### 当前观察

已确认以下实际情况：

- 早上约 `10` 点触发过风控手机号验证
- 随后重新获取同一账号/同一 Cookie 的新登录态
- 当天再次尝试继续爬取，仍然无法恢复正常抓取

### 当前结论

当前阶段可以先按以下规则理解：

1. 同一 Cookie / 同一账号一旦触发手机号验证，不是“刷新 Cookie 后立刻恢复”
2. 即使重新获取了最新 Cookie，当天继续使用仍然大概率不可行
3. 这类 Cookie 至少应按“冷却一天”处理

### 当前影响

这说明当前系统的恢复方式不能理解成：

- 触发验证
- 重新导出 Cookie
- 立刻继续跑

而应理解成：

- 一旦触发手机号验证
- 当前 Cookie / 账号当天退出生产使用
- 至少进入 `1` 天级别冷却

### 当前阶段处理方式

本阶段先将其记录为实测规则，不立即自动化实现。

后续 Phase6 设计 Cookie profile / 账号池时，这条应视为高优先级约束：

- 触发手机号验证的 Cookie profile
- 不能在当天再次投入抓取

## 2026-06-02 当前阶段目标修正：从 200 条下调到 120 条

### 当前背景

结合目前已知观察：

- 单 Cookie 正常连续抓取约 `30` 条接近风控边界
- 异常职位会明显抬高风险
- 一旦触发手机号验证，当天基本不可恢复

原先按单日 `200` 条设计，当前阶段现实性偏低。

### 当前更现实的目标

当前阶段更合理的目标应调整为：

- 找到 `3` 个可用 Cookie
- 在不触发手机号验证的前提下轮换使用
- 总量先以约 `120` 条为目标

可理解为：

- 单 Cookie 目标量级约 `30-40` 条
- `3` 个 Cookie 合计目标约 `120` 条

### 当前结论

现阶段不应再把 `200` 条作为默认可达目标，而应先按：

- `120` 条可稳定运行
- 且不触发手机号验证

作为当前更现实的阶段性目标。

### 当前阶段意义

这条目标修正意味着：

1. 后续策略设计要围绕“低风险稳定 120 条”展开
2. `200` 条不作为当前阶段默认基线
3. 先把“多 Cookie 低风险轮换”跑通，再评估是否有必要继续往上推量
